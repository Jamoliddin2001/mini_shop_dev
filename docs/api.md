# API Reference

Базовый префикс — `/api`. Все ответы — JSON. Ошибки — в едином формате `ApiError`.
Полная карта эндпоинтов и схема БД — в [Architecture](ARCHITECTURE.md); здесь — практические детали
по фазе товаров.

## Аутентификация

JWT (Bearer). Получите токен через `POST /api/auth/login`, затем шлите его в заголовке:

```
Authorization: Bearer <accessToken>
```

| Метод | Путь | Доступ | Коды |
|-------|------|--------|------|
| POST | `/api/auth/register` | public | 201, 400, 409 |
| POST | `/api/auth/login` | public | 200 (+JWT), 400, 401 |
| GET | `/api/me` | USER/ADMIN | 200, 401 |

## Формат ошибки (`ApiError`)

```json
{
  "timestamp": "2026-05-29T12:00:00Z",
  "status": 404,
  "error": "Not Found",
  "message": "Product 9999 not found",
  "path": "/api/products/9999"
}
```

Поле `violations` присутствует только при ошибках валидации (400) — массив
`{ "field": "...", "message": "..." }`; пустое опускается из JSON. Stacktrace наружу не отдаётся никогда.

## Товары (`/products`)

| Метод | Путь | Доступ | Коды |
|-------|------|--------|------|
| GET | `/api/products` | public | 200, 400 |
| GET | `/api/products/{id}` | public | 200, 404 |
| POST | `/api/products` | ADMIN | 201, 400, 401, 403, 404 |
| PUT | `/api/products/{id}` | ADMIN | 200, 400, 401, 403, 404 |
| DELETE | `/api/products/{id}` | ADMIN | 204, 401, 403, 404 |

### Изображение хранится как URL (а не файловая загрузка)

Товар содержит поле `imageUrl` (строка ≤1024). **Это осознанный выбор для мини-версии:**

- не нужны объектное хранилище (S3/MinIO), права доступа и безопасность загрузок (тип/размер/антивирус);
- проще локальный запуск и тесты — достаточно внешней ссылки;
- модель тривиально расширяется до реальной загрузки позже (см. ниже).

> **Дальнейшее масштабирование.** Для продакшена: реальная загрузка в S3/MinIO + CDN; эндпоинт
> возвращает сгенерированный URL, а в БД по-прежнему хранится `imageUrl` — контракт API не меняется.
> Зафиксировано также в [ARCHITECTURE.md → раздел 5](ARCHITECTURE.md).

### Фильтрация и пагинация (`GET /api/products`)

| Параметр | Тип | Описание |
|----------|-----|----------|
| `name` | string | Частичное совпадение по названию, регистронезависимо (`ILIKE '%...%'`). |
| `categoryId` | long > 0 | Точное совпадение по категории. |
| `minPrice` / `maxPrice` | decimal ≥ 0 | Включительные границы цены. `minPrice > maxPrice` → пустая страница (не ошибка). |
| `page` / `size` | int | Пагинация. `size` ограничен сверху 100, по умолчанию 20. |
| `sort` | string | `поле,направление`. Разрешённые поля (whitelist): `name`, `price`, `createdAt`. По умолчанию `createdAt,desc`. |

Невалидные параметры дают **400** (а не 500): нечисловая цена (`?minPrice=abc`), отрицательная
цена (`?minPrice=-1`), сортировка по неразрешённому/неизвестному полю (`?sort=bogus`).

**Пример:**

```bash
curl "http://localhost:8080/api/products?name=key&minPrice=50&sort=price,asc&page=0&size=10"
```

Ответ — конверт `PageResponse` (а не «сырой» Spring `Page`, чтобы не протекала нестабильная
структура `PageImpl`):

```json
{
  "content": [
    {
      "id": 2, "name": "Mechanical Keyboard", "description": "Hot-swappable RGB mechanical keyboard",
      "price": 89.50, "imageUrl": "https://example.com/img/keyboard.jpg",
      "categoryId": 1, "categoryName": "Electronics", "createdAt": "2026-05-29T10:00:00Z"
    }
  ],
  "page": 0, "size": 10, "totalElements": 1, "totalPages": 1
}
```

### Создание / изменение (ADMIN)

Тело запроса (`POST` / `PUT` — полная замена):

```json
{
  "name": "Wireless Mouse",
  "description": "Ergonomic 2.4GHz mouse",
  "price": 24.99,
  "imageUrl": "https://example.com/img/mouse.jpg",
  "categoryId": 1
}
```

Правила валидации: `name` обязателен (≤255); `price` обязателен, ≥0, до 10 целых + 2 дробных
(под `NUMERIC(12,2)`); `imageUrl` — корректный URL (≤1024); `categoryId` — положительный и
**должен ссылаться на существующую категорию**, иначе `404`.

```bash
# Логин админа -> токен
TOKEN=$(curl -s http://localhost:8080/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@shop.local","password":"Admin123!"}' | jq -r .accessToken)

# Создание товара
curl -X POST http://localhost:8080/api/products \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"name":"Wireless Mouse","price":24.99,"imageUrl":"https://example.com/img/mouse.jpg","categoryId":1}'
```

- Без токена → `401`; токен с ролью `USER` → `403`; админ → `201` (+ заголовок `Location`).
- `DELETE /api/products/{id}` → `204`; несуществующий id → `404`.

| Метод | Путь | Доступ | Коды |
|-------|------|--------|------|
| GET | `/api/categories` | public | 200 |

Возвращает полный список категорий (без пагинации — множество маленькое и ограниченное;

```json
[ { "id": 1, "name": "Electronics" }, { "id": 2, "name": "Books" } ]
```

## Корзина (`/cart`)

Серверная корзина, **одна на пользователя**. Все эндпоинты требуют аутентификации; пользователь
всегда работает только со своей корзиной — `userId` берётся из JWT, а не из тела/пути. Аноним → `401`.

| Метод | Путь | Доступ | Коды |
|-------|------|--------|------|
| GET | `/api/cart` | USER/ADMIN | 200, 401 |
| POST | `/api/cart/items` | USER/ADMIN | 200, 400, 401, 404 |
| DELETE | `/api/cart/items/{productId}` | USER/ADMIN | 200, 401, 404 |

- Корзина создаётся **лениво** при первом добавлении; `GET` для нового пользователя вернёт пустую
  корзину (ничего не пишет в БД).
- **Добавление суммирует количество:** повторный `POST` того же товара увеличивает `quantity`
  существующей строки (инвариант БД `UNIQUE(cart_id, product_id)` — одна строка на товар).
- `productId` несуществующего товара → `404`; `quantity < 1` → `400`.
- Сумма корзины считается по **текущей** цене товара — корзина это «живое предложение»; фиксация
  цены происходит только при оформлении заказа (см. ниже).

```bash
TOKEN=$(curl -s http://localhost:8080/api/auth/login -H 'Content-Type: application/json' \
  -d '{"email":"user@shop.local","password":"Secret123"}' | jq -r .accessToken)

# Добавить 2 шт. товара #1 в корзину
curl -X POST http://localhost:8080/api/cart/items \
  -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"productId":1,"quantity":2}'
```

```json
{
  "items": [
    { "productId": 1, "productName": "Wireless Headphones", "unitPrice": 129.99, "quantity": 2, "lineTotal": 259.98 }
  ],
  "totalAmount": 259.98,
  "totalItems": 2
}
```

## Заказы (`/orders`)

Оформление заказа из корзины и просмотр истории. Все эндпоинты требуют аутентификации; пользователь
видит и создаёт **только свои** заказы.

| Метод | Путь | Доступ | Коды |
|-------|------|--------|------|
| POST | `/api/orders` | USER/ADMIN | 201, 400, 401 |
| GET | `/api/orders` | USER/ADMIN | 200, 401 |
| GET | `/api/orders/{id}` | USER/ADMIN | 200, 401, 404 |

- `POST /api/orders` — **без тела**: заказ собирается из серверной корзины. Пустая/отсутствующая
  корзина → `400 Cart is empty`. Успех → `201` + заголовок `Location`, тело — созданный заказ
  (статус `NEW`). После оформления **корзина очищается**.
- `GET /api/orders` — пагинированная история (конверт `PageResponse`, по умолчанию `createdAt,desc`;
  whitelist сортировки — `createdAt`). Возвращает **сводки без строк** (для списка).
- `GET /api/orders/{id}` — детали заказа со строками. **Чужой заказ → `404`** (а не `403`): не
  раскрываем существование чужих ресурсов.

```bash
# Оформить заказ из текущей корзины
curl -X POST http://localhost:8080/api/orders -H "Authorization: Bearer $TOKEN"
```

```json
{
  "id": 10,
  "status": "NEW",
  "totalAmount": 259.98,
  "createdAt": "2026-05-30T10:00:00Z",
  "items": [
    { "productId": 1, "productName": "Wireless Headphones", "unitPrice": 129.99, "quantity": 2, "lineTotal": 259.98 }
  ]
}
```

### Фиксация цены (snapshot) — управление рисками

Строка заказа (`order_items`) хранит **снимок** `productName` и `unitPrice` на момент оформления,
а не ссылку на текущую цену товара. Сумма заказа `totalAmount` = Σ снимков. Зачем:

- **Цена изменчива** (промо, инфляция, исправление ошибки). Заказ — финансовый факт на момент
  оформления; ссылка на «живую» цену означала бы, что историческая сумма **меняется задним числом**
  при каждом редактировании прайса → испорченная отчётность и споры с клиентом.
- **Товар может быть переименован/удалён.** Снимок делает строку заказа самодостаточной — заказ
  читается даже после изменений каталога.
- **Двойная защита целостности:** FK `order_items → products` объявлен `ON DELETE RESTRICT` (нельзя
  физически удалить заказанный товар), а снимок дополнительно защищает от изменения названия/цены.
  Это осознанная денормализация: неизменяемый документ-заказ изолирован от изменяемого каталога.

### Транзакционность и параллельные заказы

- **Весь checkout — одна транзакция:** чтение корзины → создание `order` + `order_items` → очистка
  корзины атомарны (всё или ничего). Невозможны состояния «заказ есть, корзина не очищена» и наоборот.
- **Параллельные оформления одной корзины** сериализуются **пессимистичной блокировкой строки
  `cart`** (`SELECT … FOR UPDATE`): второй поток дожидается коммита первого и видит уже пустую
  корзину → `400`. Итог — из одной корзины не более одного заказа.

### Уведомление о заказе — асинхронное событие

При успешном оформлении публикуется доменное событие `OrderCreated`, которое обрабатывается
**после коммита транзакции** (`@TransactionalEventListener(AFTER_COMMIT)`) и **асинхронно**
(`@Async`, ограниченный пул потоков). Сейчас обработчик — заглушка (лог уведомления, без PII).

> **Дальнейшее масштабирование.** В продакшене это событие ушло бы в **брокер сообщений**
> (Kafka / RabbitMQ / SQS), а отдельный сервис уведомлений обрабатывал бы его независимо. Точка
> расширения изолирована в слушателе — `OrderService` менять не пришлось бы.

## База данных и миграции

Схема и индексы под каталог, корзину и заказы уже созданы в `V1__init.sql` — **фаза товаров,
корзины и заказов не добавляет новых миграций**. Релевантные индексы:

| Индекс | Назначение |
|--------|-----------|
| `idx_products_name_trgm` (GIN, `pg_trgm`) | Поиск по названию `ILIKE '%...%'`. |
| `idx_products_category_id` | Фильтр по категории. |
| `idx_products_price` | Фильтр/сортировка по диапазону цены. |
| `idx_products_created_at` (DESC) | Сортировка «сначала новые» по умолчанию. |
| `idx_cart_items_cart_id` | Загрузка строк корзины пользователя. |
| `idx_orders_user_id` | История заказов пользователя. |
| `idx_order_items_order_id` | Загрузка строк заказа. |

Ключевые ограничения целостности: `UNIQUE(cart_id, product_id)` (одна строка на товар в корзине),
`CHECK(quantity > 0)`, `CHECK(total_amount >= 0)`, `cart_items → products ON DELETE CASCADE`
(удаление товара чистит строки корзины), `order_items → products ON DELETE RESTRICT`
(заказанный товар удалить нельзя).

Категория (`@ManyToOne`, LAZY) подтягивается fetch-join'ом в запросе листинга; корзина и детали
заказа грузятся через `@EntityGraph` (строки + товары одним запросом) — это предотвращает N+1
(проверяется интеграционными тестами).
